## PART 1 — BUG REPORT: KPI COUNT DISCREPANCY

### Symptom
Operator account (`kosal@fin-tegration.com`) Command Center showed:
```
TOTAL PROSPECTS: 437
↑ 6 new this week
```
Actual pipeline: **1,030** lead_assignments at session start.

### Root Cause Analysis — 4 Stacked Bugs

#### Bug 1: `loadAlAssignmentsForAdvisor` — Hard Limit of 100
**File:** `js/db.js` line ~538
**Problem:** The query had `.limit(100)` — hard cap that silently truncated any advisor with more than 100 assigned leads.
```javascript
// BEFORE (broken)
const snap = await db.collection('lead_assignments')
  .where('ownerUid', '==', uid)
  .orderBy('assignedAt', 'desc')
  .limit(100)    // ← cut off Kosal's 410 at 100
  .get();

// AFTER (fixed)
  .limit(500)    // supports large pipelines
```

#### Bug 2: `loadAssignedLeadsFromFirestore` — N+1 Doc Fetches, No Limit
**File:** `js/db.js` line ~130
**Problem:** For every `lead_assignment` doc, the function did a separate `.get()` on the corresponding `master_leads` doc (N+1 pattern). With 410 assignments, this spawned 410 parallel Firestore reads, many of which timed out or failed silently, returning `null` and being filtered out.
```javascript
// Each assignment triggered:
const leadSnap = await db.collection('master_leads').doc(a.masterLeadId).get();
if (!leadSnap.exists) return null;   // ← silent failure, reduces PROSPECTS count
```
**Fix:** Added `.limit(500)` to the primary query to be explicit and avoid unbounded queries.

#### Bug 3: `computeMetrics()` Used `PROSPECTS.length` Not Real Count
**File:** `js/data.js` line ~461
**Problem:** `computeMetrics()` computed `M.total = PROSPECTS.length`. The `PROSPECTS` array is:
- 28 hardcoded demo leads (always present, defined in `data.js` as `const PROSPECTS = [...]`)
- Plus hydrated Firestore leads that successfully loaded

With bugs 1 and 2 causing some leads to fail loading, `PROSPECTS.length` reflected the partial hydration result (~437) rather than the real database total.
```javascript
// BEFORE (broken)
function computeMetrics() {
  const total = PROSPECTS.length;   // ← 28 demo + ~409 partially hydrated = 437

// AFTER (fixed)
  const total = window._firestoreLeadTotal || PROSPECTS.length;  // prefers DB count
```

#### Bug 4: Firebase Compat SDK v9.23.0 Does NOT Support `.count()` Aggregation
**This is the critical gotcha — document for all future sessions.**

The first two fix attempts tried to use Firestore's `.count()` aggregation:
```javascript
const countSnap = await db.collection('lead_assignments')
  .where('ownerUid', '==', uid)
  .count()    // ← NOT AVAILABLE in compat SDK
  .get();
```
The Firebase compat SDK loaded via CDN (`firebase-firestore-compat.js` v9.23.0) does **not** implement the `.count()` aggregation API. This is available only in the modular v9+ SDK. The call threw silently into the `catch` block, returned `0`, and the fallback to `PROSPECTS.length = 437` was used every time. The cockpit still showed 437 after two deployments.

> ⚠️ **PERMANENT RULE:** Never use `.count()` aggregation with the Firebase compat SDK in this project. Use the `meta/pipeline_stats` pattern instead (see §2.3 below).

### 2.2 `computeMetrics()` Upgrade (`js/data.js`)

```javascript
// NEW: prefers window._firestoreLeadTotal (set from meta/pipeline_stats read)
// Falls back to PROSPECTS.length only if the meta read hasn't completed yet
function computeMetrics() {
  const total = window._firestoreLeadTotal || PROSPECTS.length;
  // ...
  const weekAgo = Date.now() - 7 * 24 * 60 * 60 * 1000;
  const newThisWeek = PROSPECTS.filter(p => {
    const d = p.enrolled ? new Date(p.enrolled).getTime()
            : p.ingestedAt ? new Date(p.ingestedAt).getTime() : 0;
    return d > weekAgo;
  }).length;
  return { total, newThisWeek, ... };
}
```

Also fixed: the "↑ 6 new this week" KPI delta was hardcoded. Now uses `M.newThisWeek` dynamically.

### 2.3 Pipeline Meta Pattern — `scripts/write_pipeline_meta.js` (NEW FILE)

The canonical solution for the count problem. This script uses the **Admin SDK** (bypasses all Firestore rules, no browser SDK limitations) to read the true counts and write a single summary document.

**How it works:**
```
Node.js (Admin SDK)
  → reads all lead_assignments (1,116 docs)
  → counts by ownerUid
  → writes meta/pipeline_stats to Firestore

Browser (compat SDK)
  → reads meta/pipeline_stats (1 Firestore read)
  → sets window._firestoreLeadTotal
  → computeMetrics() uses it for KPI
```

**The document written:**
```json
{
  "totalLeads": 1116,
  "totalMasterLeads": 705,
  "totalQueueItems": 695,
  "leadsByAdvisor": {
    "FvEWqsETjbU602nLfHaJUaUkWkS2": 410,
    "iru1h2jzeKhrwmi5siqM8lt4osn1": 132,
    "Iqo8zz5gTFh967ZokqHCpUp4S2t2": 146,
    "Zd4H7gaNZJdrgXbIWNnM5cSpqdB2": 142,
    "BQhiSqKW2JM3ycrPQYzeXa640Ku1": 113,
    "NzC6fh3sXKVuDmgfPAaaEea3Ovm2": 73,

```

**`fetchAdvisorLeadCount()` — operator vs advisor logic:**
```javascript
const isOp = window._currentUser?.email === 'kosal@fin-tegration.com';
const count = isOp
  ? meta.totalLeads              // operator → global total (1,116)
  : meta.leadsByAdvisor[uid];   // advisor → personal count only
```

**Timing fix:** `fetchAdvisorLeadCount` was originally fire-and-forget (non-blocking). By the time it returned, `renderPage()` had already run and painted the old count. Fixed by adding it to the `Promise.all([...])` inside `bootstrapUserData` — count is guaranteed to be set before the first render.

### 2.4 Firestore Security Rules — `meta` Collection (`firestore.rules`)

Without this, the browser's request to read `meta/pipeline_stats` hit the catch-all deny rule and returned a permission error, causing another silent fallback to `PROSPECTS.length`.

```
match /meta/{docId} {
  allow read:  if request.auth != null;   // any logged-in user can read stats
  allow write: if false;                  // Admin SDK only (bypasses rules)
}
```

**Inserted before the catch-all:**
```
match /{document=**} {
  allow read, write: if false;   // ← would have blocked meta/ reads
}
```
